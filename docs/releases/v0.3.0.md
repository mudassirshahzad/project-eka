# Release v0.3.0 — Business Use Cases

> **Status: Work In Progress**

## Overview

P02.2 implements the Business Use Cases layer — single-responsibility classes that orchestrate the Domain layer through Application Services. Use cases are the stable entry-point surface that REST controllers (P05) will call and that enforce business validation before delegating to Application Services.

## What Changed

### Document Use Cases (`application.document`)
- `UploadDocumentUseCase` — validates filename not blank and format-filename consistency before registering
- `GetDocumentUseCase` — retrieves a document scoped to a tenant
- `ListDocumentsUseCase` — lists documents by tenant or by owner (two `execute` overloads)
- `DeleteDocumentUseCase` — soft-deletes a document

### Conversation Use Cases (`application.conversation`)
- `CreateConversationUseCase` — validates title (not blank, ≤ 500 chars) before creating
- `GetConversationUseCase` — retrieves a conversation scoped to a user
- `ListConversationsUseCase` — lists conversations by user and tenant
- `DeleteConversationUseCase` — guards deletion: rejects if an active `ChatSession` exists (cross-service coordination)

### Chat Session Use Cases (`application.chat`)
- `StartChatSessionUseCase` — validates modelId not blank before starting a session
- `CompleteChatSessionUseCase` — transitions an active session to COMPLETED
- `TimeoutChatSessionUseCase` — transitions an active session to TIMED_OUT

### Knowledge Query Use Cases (`application.query`)
- `CreateKnowledgeQueryUseCase` — validates queryText (not blank, ≤ 10 000 chars) before submitting
- `GetKnowledgeQueryUseCase` — retrieves a query by ID
- `ListKnowledgeQueriesUseCase` — lists queries by user and tenant

### User Use Cases (`application.user`)
- `RegisterUserUseCase` — validates email format (contains `@` with chars on both sides), passwordHash not blank, roles not empty
- `GetUserUseCase` — retrieves a user by ID
- `DeactivateUserUseCase` — deactivates a user within a tenant

## Architecture Notes

- Use cases are `@Service` beans in the same package as their Application Service.
- Write use cases are `@Transactional`; read use cases are `@Transactional(readOnly = true)`.
- Each use case has exactly one public `execute()` method.
- `DeleteConversationUseCase` is the only use case that coordinates two Application Services (`ConversationApplicationService` + `ChatSessionApplicationService`), demonstrating the layer's cross-service coordination role.
- No infrastructure classes are referenced; the boundary is enforced by the existing ArchUnit rules.
