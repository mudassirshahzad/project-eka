# Release v0.3.0 — Application Layer

> **Status: Ready for Release**

## Overview

v0.3.0 completes the Application Layer (P02) — the orchestration tier that sits between the frozen Domain model and the Infrastructure adapters. This release establishes the stable entry-point surface that future REST controllers (P05) will call.

Three sub-phases were delivered:

| Sub-phase | Scope |
|---|---|
| P02.1 – Application Layer Foundation | Services, ports, events, commands, exceptions |
| P02.2 – Business Use Cases | 17 single-responsibility use case classes |
| P02.3 – Validation & Readiness | Architecture review, ArchUnit hardening, 38 new unit tests |

---

## Features Completed

### P02.1 — Application Layer Foundation

**Ports**
- `DomainEventPublisher` — outbound port; `SpringDomainEventPublisher` adapter wraps Spring `ApplicationEventPublisher`

**Exceptions**
- `ApplicationException` (base), `ResourceNotFoundException`, `DuplicateResourceException`

**Domain Events (12)** in `application.event`:
- `DocumentRegisteredEvent`, `DocumentDeletedEvent`
- `ConversationCreatedEvent`, `ConversationDeletedEvent`, `MessageAddedEvent`
- `ChatSessionStartedEvent`, `ChatSessionCompletedEvent`, `ChatSessionTimedOutEvent`
- `QuerySubmittedEvent`
- `UserRegisteredEvent`, `UserActivatedEvent`, `UserDeactivatedEvent`

**Application Services (5)** — `@Service @Transactional`, `readOnly=true` overrides on queries:
- `DocumentApplicationService` — register, get, list (tenant / owner), updateMetadata, delete
- `ConversationApplicationService` — create, get, list, addUserMessage, rename, delete
- `ChatSessionApplicationService` — start (at-most-one-active invariant), recordTurn, complete, timeout, find, list
- `QueryApplicationService` — submit, get, listByUser
- `UserApplicationService` — register (duplicate-email guard), get, findByEmail, activate, deactivate, changePassword, assignRole, removeRole

**Commands (13 records):** `RegisterDocumentCommand`, `UpdateDocumentMetadataCommand`, `DeleteDocumentCommand`, `CreateConversationCommand`, `AddUserMessageCommand`, `RenameConversationCommand`, `StartChatSessionCommand`, `RecordTurnCommand`, `SubmitQueryCommand`, `RegisterUserCommand`, `ChangePasswordCommand`, `AssignRoleCommand`, `RemoveRoleCommand`

---

### P02.2 — Business Use Cases

17 `@Service` use-case classes — single `execute()` method each, validate inputs before delegating to Application Services.

**Document** (`application.document`):
- `UploadDocumentUseCase` — validates filename not blank; enforces format-filename extension consistency
- `GetDocumentUseCase`, `ListDocumentsUseCase` (tenant + owner overload), `DeleteDocumentUseCase`

**Conversation** (`application.conversation`):
- `CreateConversationUseCase` — title not blank, ≤ 500 characters
- `GetConversationUseCase`, `ListConversationsUseCase`
- `DeleteConversationUseCase` — **cross-service guard**: rejects if an active `ChatSession` exists for the conversation

**Chat Session** (`application.chat`):
- `StartChatSessionUseCase` — validates modelId not blank
- `CompleteChatSessionUseCase`, `TimeoutChatSessionUseCase`

**Knowledge Query** (`application.query`):
- `CreateKnowledgeQueryUseCase` — queryText not blank, ≤ 10 000 characters
- `GetKnowledgeQueryUseCase`, `ListKnowledgeQueriesUseCase`

**User** (`application.user`):
- `RegisterUserUseCase` — validates email format (`@` with characters on both sides), passwordHash not blank, at least one role
- `GetUserUseCase`, `DeactivateUserUseCase`

### P02.3 — Validation & Readiness

**Architecture**
- Confirmed: zero `infrastructure.*` imports in `application.*` layer
- Confirmed: Domain has zero Spring / JPA imports
- Removed `allowEmptyShould(true)` from both ArchUnit application-layer rules — rules now strictly enforce the full application layer (previously suppressed because the layer was empty)

**Tests added** — 6 use-case unit test classes (Mockito, no Spring context):

| Test class | Tests | What is covered |
|---|---|---|
| `UploadDocumentUseCaseTest` | 8 | null cmd/tenantId/format, blank/null filename, format-filename mismatch, valid + unknown extension |
| `CreateConversationUseCaseTest` | 6 | null cmd/title/userId, blank title, title > 500 chars, title = 500 chars (boundary), valid |
| `DeleteConversationUseCaseTest` | 4 | null conversationId/userId, active session blocks delete, no active session allows delete |
| `StartChatSessionUseCaseTest` | 5 | null cmd/conversationId, null/blank modelId, valid |
| `CreateKnowledgeQueryUseCaseTest` | 6 | null cmd/userId, null/blank queryText, > 10 000 chars, valid |
| `RegisterUserUseCaseTest` | 9 | null cmd/tenantId, null/blank email, no `@`, `@` at start, `@` at end, blank password, empty roles, valid |

## Known Limitations

| Limitation | Impact | Resolution |
|---|---|---|
| No integration tests | Application services are tested through unit tests with mocked ports only | Will be addressed in P05 with full stack tests against Testcontainers |
| Email validation is structural only (contains `@` with chars on both sides) | Accepts technically invalid addresses | Sufficient for internal trust boundary; full RFC validation belongs in REST API layer (P05) |
| `ListDocumentsUseCase` has two execute overloads (`execute` + `executeByOwner`) | Minor deviation from strict single-method use case pattern | Acceptable: both list the same aggregate with optional owner scope |
| Document status transitions (PARSING → CHUNKING → EMBEDDING → INDEXED) are domain methods that have no application-layer use cases yet | No business operations for ingestion lifecycle | Intentional: P03 will introduce `DocumentIngestionService` that orchestrates these transitions |

---

## Technical Debt

None introduced in this release. The application layer is consistent with the hexagonal architecture:
- Use cases → Application Services → Domain Ports → Infrastructure
- Zero framework leakage into Domain
- ArchUnit rules enforce boundaries at build time
