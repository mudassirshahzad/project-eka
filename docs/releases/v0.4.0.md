# Release v0.4.0 — Document Ingestion Pipeline

> **Status: Work In Progress — P03.1, P03.2, and P03.3 complete, P04 pending**

## Overview

v0.4.0 delivers the complete Document Ingestion Pipeline (P03). The goal is to transform uploaded raw files into indexed, queryable knowledge. The pipeline is divided into three stages:

| Sub-phase | Scope | Status |
|---|---|---|
| P03.1 – Document Parsing & Storage | Upload → Validation → Storage → Tika Parsing → Metadata Extraction | ✅ Done |
| P03.2 – Chunking & Embedding | Parsed text → SlidingWindow chunks → Ollama embeddings → DB (EMBEDDING state) | ✅ Done |
| P03.3 – Weaviate Indexing | Embedded chunks → Weaviate → INDEXED state | ✅ Done |

## Features Completed

### P03.1 — Document Parsing & Storage

**Domain — `SupportedFormat` extended**
- Added `DOC`, `JSON`, `XML` to enum
- Added `fromMimeType(String mimeType)` — maps IANA MIME types to `SupportedFormat`; trims charset parameters; falls back to declared format when unrecognised
- `isRichText()` now includes `DOC`
- `fromFilename()` handles `.doc`, `.json`, `.xml`

**Domain — new value objects**
- `ParsingStatus` enum: `SUCCESS`, `PARTIAL` (blank extracted text)
- `ParsedMetadata` record: `title`, `author`, `description`, `pageCount`, `characterCount`
- `ParsedDocument` record: `extractedText`, `metadata`, `detectedFormat`, `status`, `parsedAt`

**Domain — new ports**
- `FileStorage` — `store(relativePath, bytes)`, `load(relativePath)`, `delete(relativePath)`
- `DocumentParser` — `parse(bytes, format)` returning `ParsedDocument`

**Domain — `Document` extended**
- New field `parsedTextPath` with `assignParsedTextPath(String)` and `getParsedTextPath()`
- `reconstitute()` takes `parsedTextPath` as its 9th parameter (after `rawContentPath`)

**Infrastructure — storage**
- `LocalFileStorageAdapter implements FileStorage`
  - Root: `app.storage.document-root` (default: `${DOCUMENT_STORAGE_ROOT:/data/documents}`)
  - Path structure: `{tenantId}/{documentId}/raw/{filename}` for raw; `{tenantId}/{documentId}/parsed.txt` for text
  - Creates directories on demand; returns relative path from document root
- `FileStorageException` — unchecked, wraps `IOException`

**Infrastructure — parsing**
- `TikaDocumentParserAdapter implements DocumentParser`
  - Uses `AutoDetectParser` (thread-safe singleton within the bean)
  - Sets `Parser.class` in `ParseContext` to enable recursive embedded-document extraction
  - Extracts `title`, `author`, `description`, `pageCount` (`meta:page-count`) from Tika metadata
  - Returns `ParsingStatus.PARTIAL` when extracted text is blank (scanned/encrypted documents)
  - Detects actual MIME type and maps to `SupportedFormat` via `fromMimeType()`; falls back to declared format
- `DocumentParsingException` — unchecked, wraps Tika/SAX/IO exceptions

**Infrastructure — persistence**
- `DocumentEntity` — `parsed_text_path VARCHAR(1000)` column mapping added
- `DocumentPersistenceMapper` — `toDomain()`, `toEntity()`, `updateEntity()` all propagate `parsedTextPath`

**Application — new files**
- `UploadDocumentCommand` record — adds `byte[] content` to the existing metadata fields
- `DocumentParsedEvent` — carries `documentId`, `tenantId`, `detectedFormat`, `parsedTextPath`

**Application — updated files**
- `DocumentApplicationService.updateDocument(Document)` — persists in-memory domain state changes via `documentRepository.save()`
- `UploadDocumentUseCase` — full 8-step pipeline:
  1. Input validation (filename, format, content not null/empty, format–extension consistency)
  2. `documentService.registerDocument()` → DB INSERT with `PENDING` status; publishes `DocumentRegisteredEvent`
  3. `document.startParsing()` → in-memory transition to `PARSING`
  4. `documentParser.parse()` → Tika extraction (throws `DocumentParsingException` → `@Transactional` rollback, no files written)
  5. `fileStorage.store(rawPath, content)` → raw bytes on disk
  6. `fileStorage.store(parsedPath, text.getBytes(UTF-8))` → extracted text on disk
  7. `document.assignContentPath()` + `assignParsedTextPath()` + `documentService.updateDocument()` → DB UPDATE
  8. `eventPublisher.publish(DocumentParsedEvent)` → signals P03.2 chunking pipeline

**Database — Flyway V015**
- `ALTER TABLE documents ADD COLUMN parsed_text_path VARCHAR(1000)`
- `chk_documents_format` constraint updated to include `'DOC','JSON','XML'`

### P03.2 — Chunking & Embedding Pipeline

**Domain — new port**
- `EmbeddingProvider` — `embed(List<String>)`, `modelName()`, `dimension()` in `domain.chunk`

**Domain — `Chunk` extended**
- Fields: `embeddingModel`, `embeddingDimension`, `embeddedAt`
- `assignEmbeddingProvenance(model, dimension, embeddedAt)` — one-way write, guards with `requireNonNull`
- `isEmbedded()` — checks `embeddingModel != null` (semantics: has provenance, regardless of Weaviate state)
- `reconstitute()` updated to 11-parameter signature with provenance fields before `createdAt`

**Domain — `Document` extended**
- `updateChunkCount(int count)` — validates non-negative, updates `chunkCount` and `updatedAt`

**Application — new value types**
- `TextSegment` record: `content`, `startOffset`, `endOffset`, `tokenCount`, `sequenceNumber`
- `EmbeddedChunk` record: `Chunk chunk`, `float[] embedding`

**Application — new interfaces**
- `ChunkingStrategy` — `name()` + `chunk(String text) → List<TextSegment>`

**Application — new services**
- `SlidingWindowChunkingStrategy implements ChunkingStrategy` — token-span sliding window; paragraph-boundary snapping; `@Value` constructor injection for `chunk-size` (default 512) and `chunk-overlap` (default 64); guaranteed forward progress; returns immutable list
- `ChunkingService` — delegates to `ChunkingStrategy`, wraps `TextSegment`s into `Chunk` domain objects with `ChunkMetadata`
- `EmbeddingService` — batch processing with configurable `embedding-batch-size` (default 32); manual exponential backoff (100ms → 1000ms, 3 retries); assigns provenance in-memory before DB write
- `ChunkApplicationService` — `@Transactional`; `saveAll()` calls `chunkRepository.saveAll()`, publishes `ChunkCreatedEvent` + `ChunkEmbeddedEvent` per chunk
- `IngestionBenchmarkService` — times chunk → embed → persist phases, returns `BenchmarkReport`

**Application — new events**
- `ChunkCreatedEvent` — `chunkId`, `documentId`, `tenantId`, `sequenceNumber`
- `ChunkEmbeddedEvent` — `chunkId`, `documentId`, `tenantId`, `embeddingModel`, `embeddingDimension`

**Application — `UploadDocumentUseCase` extended**
- Now takes `ChunkingService`, `EmbeddingService`, `ChunkApplicationService` via constructor injection
- Extended to 13-step pipeline: parse → store files → chunk (CHUNKING) → embed (EMBEDDING) → saveAll → updateChunkCount → updateDocument
- Document reaches `EMBEDDING` status after successful ingestion (P03.3 will transition to `INDEXED`)

**Infrastructure — embedding**
- `OllamaEmbeddingProvider implements EmbeddingProvider` — wraps Spring AI `EmbeddingModel`; `@Value` for model name; delegates `dimension()` to `embeddingModel.dimensions()`
- `EmbeddingException` — unchecked, wraps Spring AI failures

**Infrastructure — persistence**
- `ChunkEntity.vectorId` — `nullable = false` removed; now nullable until Weaviate indexing (P03.3)
- `ChunkPersistenceMapper` — `toDomain()` + `toEntity()` propagate `embeddingModel`, `embeddingDimension`, `embeddedAt`

**Database — Flyway V016**
- `ALTER TABLE chunks ALTER COLUMN vector_id DROP NOT NULL`
- PostgreSQL UNIQUE constraint retained — NULLs are treated as distinct, so multiple un-indexed chunks coexist safely

## Architecture Notes

- **No hexagonal boundary violations**: `FileStorage` and `DocumentParser` are domain-layer interfaces; `UploadDocumentUseCase` imports only from `domain.*` and `application.*` — no infrastructure imports
- **Transaction safety**: Tika parse runs BEFORE any file I/O. Parse failure = `@Transactional` rollback = no DB record, no orphaned files
- **Orphan risk (accepted)**: If `fileStorage.store()` succeeds but `updateDocument()` fails, the raw file is orphaned. Cleanup policy deferred to P03.3 (dead-letter queue / reconciliation job)
- **State after P03.1**: Documents remain in `PARSING` status with `parsedTextPath` set. P03.2 will query PARSING documents with non-null `parsedTextPath` and call `startChunking()`
- **ArchUnit (8 rules)**: All pass — infrastructure adapters import domain ports only; application layer has zero infrastructure imports

## Known Limitations

| Limitation | Impact | Resolution |
|---|---|---|
| Password-protected / scanned PDFs return `PARTIAL` | No text extracted; zero chunks produced | Out of scope for P03; OCR is a future P09 concern |
| File storage is local filesystem only | Not suitable for distributed deployment | `FileStorage` port makes S3/GCS adapter a one-file change |
| Ingestion is synchronous | `UploadDocumentUseCase` blocks until Weaviate indexing completes; large docs may be slow | Async pipeline deferred to post-P03 |
| `parseTime` in `BenchmarkReport` is always `Duration.ZERO` | Parsing is done before `IngestionBenchmarkService` is called | By design — benchmark measures chunk+embed+index+persist only |
| Double-embedding (technical debt) | P03.2 embeds via `OllamaEmbeddingProvider`; P03.3 `WeaviateVectorStoreAdapter` re-embeds internally via Spring AI when calling `springVectorStore.add()` | Spring AI 1.0.0 `WeaviateVectorStore.doAdd()` always calls `embeddingModel.embed()` — cannot pass pre-computed vectors through the `VectorStore` abstraction |
| Transaction scope spans Weaviate HTTP call | `UploadDocumentUseCase` is `@Transactional`; Weaviate HTTP call happens while DB connection is held | Acceptable for synchronous pipeline; deferred to async redesign |

**Application — new events**
- `DocumentIndexedEvent` — `documentId`, `tenantId`, `chunkCount`
- `ChunkIndexedEvent` — `chunkId`, `documentId`, `tenantId`, `vectorId`

**Application — new value type**
- `IngestionValidationResult` record — `valid`, `expectedCount`, `indexedCount`, `violations`

**Application — new services**
- `DocumentIndexingService` — batch-indexes chunks into Weaviate via `VectorStore` port; deletes stale vectors before re-indexing; assigns vectorIds via mutation; persists chunks to DB; publishes `ChunkIndexedEvent` per chunk; returns updated chunks
- `IngestionValidationService` — validates: count match, no null vectorIds, no duplicates, all chunks have embedding provenance; returns `IngestionValidationResult`

**Application — updated files**
- `UploadDocumentUseCase` — extended from 13-step to 15-step pipeline:
  - Step 12: `documentIndexingService.index(savedChunks)` → vectorIds assigned + DB updated
  - Step 13: `document.markIndexed(indexedChunks.size())` → INDEXED state
  - Step 14: `documentService.updateDocument(document)` → DB UPDATE
  - Step 15: publishes `DocumentParsedEvent` + `DocumentIndexedEvent`
- `DeleteDocumentUseCase` — extended to delete Weaviate vectors + PostgreSQL chunks before soft-deleting document (delete synchronization)
- `BenchmarkReport` — added `indexTime`, `vectorsPerSecond()` fields + `index=` in `summary()`
- `IngestionBenchmarkService` — added `DocumentIndexingService` dependency; measures `indexTime`

## Design Decisions (P03.3)

| Decision | Rationale |
|---|---|
| `VectorStore.index(List<Chunk>)` assigns vectorIds by mutation | Avoids a separate DTO record; domain Chunk is already the result type; matches the re-index flow where chunks must be mutated before DB save |
| Two-phase DB save | Chunks saved after P03.2 with provenance and `vectorId=null`; updated again after P03.3 with vectorId. Provides a recovery point if Weaviate fails. |
| `Chunk.clearVectorId()` added | Re-index flow requires deleting the stale Weaviate vector, clearing the chunk's vectorId, then calling `assignVectorId()`. Without `clearVectorId()` the guard in `assignVectorId()` would throw. |
| `DeleteDocumentUseCase` owns vector deletion | Keeps delete synchronization co-located with the use case rather than distributed across services. No new port methods were needed beyond `VectorStore.deleteByIds()`. |
